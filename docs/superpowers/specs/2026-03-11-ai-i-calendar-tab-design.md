# AI-I Calendar Tab Design

**Дата:** 2026-03-11

## Цель

Добавить `AI-I` как новую вкладку внутри `/calendar`, не ломая уже доставленный редизайн, где `Inbox + AI` объединены на домашнем маршруте `/`.

## Текущее состояние

- `/` рендерит `HomeWorkspace` и является новой объединённой поверхностью для `Inbox + AI`.
- `/calendar` принимает только календарные вкладки и редиректит legacy `tab=inbox` и `tab=ai` обратно на `/`.
- Локальный diff `AI-I` вырос из старой базы и вместе с новой вкладкой случайно возвращает `Inbox` и `AI` обратно в тулбар `/calendar`.

## Целевое поведение

- `/calendar` показывает вкладки `AI-I`, `Календарь`, `Канбан`, `Заметки`.
- Ссылка `AI` на `/` остаётся доступной из тулбара календаря как переход в объединённый home workspace.
- `tab=ai-i` поддерживается в URL state календаря.
- `tab=inbox` и `tab=ai` остаются legacy-значениями и продолжают редиректиться на `/`.
- `AI-I` использует локальную runtime-поверхность внутри `/calendar`.
- Плавающий widget по умолчанию остаётся на legacy surface `AI`.

## Подход

1. Оставить домашний `Inbox + AI` без изменений.
2. Добавить `AI-I` в registry AI-surfaces как единственную календарную вкладку.
3. Очистить `Calendar2` от случайно возвращённых `inbox`/`ai` табов.
4. Перепривязать тесты к новому контракту:
   - `AI-I` виден в календаре
   - legacy `inbox/ai` в календарь не возвращаются
   - `AI` остаётся отдельной домашней surface

## Риски

- Если оставить `inbox/ai` в `Calendar2Tab`, можно незаметно сломать legacy redirect flow.
- Если показывать `AI` как календарную вкладку, пользователь увидит две разные AI-поверхности в одном месте, что противоречит уже влитому редизайну.
